# データ消費前のバリデーション

[日本語](data-validation-ja.md) | [English](data-validation.md)

このパターンでは、FSx for ONTAP 上のレプリケーションファイルを **未加工のレプリケーションデータ** として扱います。Amazon Quick / QuickSight や AI サービスにデータを公開する前に、オペレーターはデータ品質を検証すべきです。

> **スコープ注記**: このパターンにおける「バリデーション済み」とは、デモまたは PoC のために最低限の鮮度、スキーマ、利用境界チェックをパスしたことを意味します。エンタープライズデータ品質認定を意味するものではありません。このバリデーションレイヤーは、エンタープライズデータ品質プラットフォームやトランザクションデータレイクの代替を意図していません。

> 参考: AWS は [公式 Glue ETL チュートリアル](https://docs.aws.amazon.com/fsx/latest/ONTAPGuide/tutorial-transform-data-with-glue.html)を提供しており、S3 Access Points 経由で FSx から生データを読み取り、変換・検証し、同じ FSx ボリュームにキュレーション済み出力を書き戻す方法を示しています。

## データの状態

| 状態 | 場所 | 説明 |
|------|------|------|
| 正本 | オンプレミス ONTAP | 権威あるビジネスデータ |
| 未加工レプリケーションコピー | FSx for ONTAP（SnapMirror 後） | 正確なレプリカ、消費用に検証されていない |
| バリデーション済みデータセット | FSx for ONTAP（検証後） | 鮮度と品質チェックをパス |
| 消費レイヤー | Amazon Quick / QuickSight | ビジネス向けの可視化と分析 |

## 最低限のバリデーションチェック

Quick / QuickSight / Bedrock / AI サービスでデータを消費する前に:

- [ ] `source_of_record_timestamp` が存在し、パース可能
- [ ] `replicated_at` タイムスタンプが存在
- [ ] 期待されるスキーマの必須カラムまたはフィールドが存在
- [ ] ファイル数が期待範囲内（ゼロでなく、異常に多くもない）
- [ ] オブジェクトサイズがゼロでない
- [ ] スキーマバージョンが認識済み
- [ ] データ分類が割り当て済み
- [ ] `dashboard_refreshed_at` が `replicated_at` より新しい
- [ ] 重複レコードが許容範囲内
- [ ] 機密フィールドが識別され処理済み

## バリデーション実装オプション

| アプローチ | 使用場面 | AWS サービス |
|-----------|---------|-------------|
| 軽量（デモ/PoC） | ファイル数少、シンプルなスキーマ | SnapMirror 完了後に AWS Lambda をトリガー |
| 中規模（本番 PoC） | スキーマ検証、フィルタリング | S3 AP 経由で読み取る AWS Glue ETL ジョブ |
| フル（本番） | 複雑な変換、パーティショニング | AWS Glue + Glue Data Quality |

## 機能境界におけるバリデーション済み vs 未加工

| 領域 | このパターンが提供するもの | このパターンが提供しないもの |
|------|------------------------|-----------------------------|
| データバリデーション | 消費前の基本的な鮮度・スキーマ検証 | フルトランザクションデータレイク管理 |
| データ品質 | デモレベルのバリデーションチェック | エンタープライズグレードのデータ品質フレームワーク |
| キュレーション出力 | オプションの FSx 上のバリデーション済みデータセット | 必須のデータレイク移行 |
