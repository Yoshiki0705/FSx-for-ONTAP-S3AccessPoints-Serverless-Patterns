# Amazon Quick Desktop × AgentCore MCP Gateway Setup Guide

> ⚠️ **This document describes a PoC-only configuration.** Gateway authentication is NONE (no auth) — do not connect to production data. For production, apply CUSTOM_JWT + Authorization Policy or VPC-internal deployment + Security Groups.

> **Audience**: SAs / Engineers who want to analyze EDA/business data on FSx for ONTAP via MCP from Amazon Quick
>
> **Workshop**: [FSx for ONTAP S3 Access Points Workshop (Module 09)](https://catalog.us-east-1.prod.workshops.aws/workshops/9cd82e0b-8348-456b-932a-818b9e5825a1/en-US/09-agentcore)
>
> **Verified versions**: Quick Desktop v0.1000.1495 / AgentCore Gateway 2026-07 GA

---

## Overview

From the Amazon Quick Desktop app, you can read and search files on FSx for ONTAP S3 Access Points in real time via AgentCore MCP Gateway.

```
Quick Desktop (MCP Client, stdio via mcp-remote)
    ↓ HTTP Streamable
AgentCore MCP Gateway (NONE auth — PoC)
    ↓ Lambda Invoke
MCP Tools Lambda (list_files / read_file / search_files)
    ↓ S3 API
FSx for ONTAP S3 Access Point
    ↓
NFS/SMB Volume (EDA simulation logs, etc.)
```

## Detailed Steps

### Step 1: Infrastructure Deployment

#### Option A: One-click script (recommended)

```bash
./scripts/deploy-agentcore-mcp.sh \
  --s3ap-alias <your-s3ap-alias> \
  --s3ap-name <your-s3ap-name> \
  --stack-name agentcore-mcp-eda
```

The script automatically:
1. Deploys CloudFormation stack (Lambda + IAM Roles)
2. Creates AgentCore Gateway (NONE auth, PoC)
3. Registers Lambda target (with tool schema)
4. Verifies 3 tools are detected via `tools/list`
5. Generates Quick Desktop MCP config JSON at `.private/mcp-agentcore-quick.json`

#### Option B: Manual CloudFormation deployment

```bash
aws cloudformation deploy \
  --template-file infrastructure/agentcore-mcp-gateway/template.yaml \
  --stack-name agentcore-mcp-eda \
  --parameter-overrides \
    S3AccessPointAlias=<your-alias> \
    S3AccessPointName=<your-name> \
  --capabilities CAPABILITY_NAMED_IAM \
  --region us-east-1
```

After deployment, manually execute the script's Steps 3–5 for Gateway creation.

### Step 2: Quick Desktop Sign-in

**Email-based** is the simplest method.

1. Launch Quick Desktop → Region: your Quick account's region
2. Select **"Continue with 📧" (Email)**
3. Enter your QuickSight registration email
4. Click the email verification link → "Allow access" in browser
5. Automatically returns to Desktop

> **Common confusion**: IAM username ≠ QuickSight username. Verify with:
> ```bash
> aws quicksight list-users --aws-account-id $(aws sts get-caller-identity --query Account --output text) \
>   --namespace default --region ap-northeast-1 --query 'UserList[*].{User:UserName,Email:Email}'
> ```

### Step 3: Add MCP Server (Import method)

> **Verified**: Import is the only stable method (as of 2026-07).
> Local / Remote direct addition may not persist due to a bug.

1. Settings → Capabilities → **Connectors** tab
2. **+ Create** → **MCP server** → **Import** tab
3. Enter the config file path:

```
<project-root>/.private/mcp-agentcore-quick.json
```

JSON contents (auto-generated by the deploy script):

```json
{
  "mcpServers": {
    "EDA Log Analyzer": {
      "command": "npx",
      "args": ["-y", "mcp-remote@latest", "https://<gateway-id>.gateway.bedrock-agentcore.us-east-1.amazonaws.com/mcp"],
      "env": {},
      "disabled": false
    }
  }
}
```

4. **Load file** → "Kiro / Claude Code — 1 server found"
5. **Import 1 server**
6. Confirmation dialog → **Add server**
7. Success: "**EDA Log Analyzer — 3 tools · 3 write · Connected**"

### Step 4: Verification

| Query | Tool | Expected Result |
|-------|------|-----------------|
| `List files in eda-regression/simulation/` | list_files | Table of 50 files |
| `Read JOB_00005's log and tell me the result` | read_file | FAIL + UVM_FATAL root cause |
| `Search for logs containing "JOB_0004" with preview` | search_files | 10 hits (7 PASS / 3 FAIL) + details |

## Lessons Learned

### Working configurations

| Component | Configuration | Notes |
|---|---|---|
| Gateway auth | **NONE** | PoC only. Production: VPC + SG |
| MCP addition method | **Import** (JSON file) | Local/Remote has bugs |
| MCP transport | **mcp-remote** (stdio proxy) | Remote direct is unstable |
| Lambda region | **ap-northeast-1** (same as FSx for ONTAP) | Same-region verified 2026-07-22 |

### Non-working configurations (as of 2026-07)

| Configuration | Cause |
|---|---|
| Web console MCP connector Step 2 | UI bug "Fix highlighted fields" |
| Desktop → Remote/Local direct add | Not persisted |
| Gateway CUSTOM_JWT + Desktop | 403 (authorization policy mismatch) |
| Gateway AWS_IAM + Desktop | SigV4 not supported |

## Production Migration

| # | Improvement | Method |
|---|-------------|--------|
| 1 | Gateway auth | NONE → CUSTOM_JWT + authorization policy |
| 2 | Network | VPC-internal deployment + Security Group |
| 3 | Lambda | Inline → ZIP package |
| 4 | Monitoring | CloudWatch Metrics + X-Ray |
| 5 | Data region | Migrate Lambda to ap-northeast-1 (awaiting AgentCore support) |

> **Data residency note**: ~~In the current PoC configuration, Lambda (us-east-1) retrieves file content from S3 AP (ap-northeast-1), meaning data crosses region boundaries.~~ **Verified 2026-07-22: AgentCore Gateway + Lambda CAN be deployed in ap-northeast-1.** All components can run in the same region, eliminating cross-region data transfer.
